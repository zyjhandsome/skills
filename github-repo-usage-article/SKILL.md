# GitHub 仓库使用示例文章(房屋风格)

把一个 GitHub 开源仓库写成一篇**可直接照做**的中文使用示例长文。读者从「完全的小白(大一新生)」到「想快速选型的工程师」都能各取所需。

> 本 Skill **自包含、可移植**:分享给其他同学或其他 Agent 时,无需访问任何外部文件即可使用。
> - 骨架模板:[TEMPLATE.md](TEMPLATE.md)(逐节占位符,复制即用)
> - **黄金范例(整篇成稿)**:[EXAMPLE.md](EXAMPLE.md)(《Codebase Memory MCP 使用示例》全文,内嵌于本目录)
>
> **首次写作前,务必先读一遍 [EXAMPLE.md](EXAMPLE.md) 体会房屋风格**(节序、案例五件套、callout、语气、中文化、校验脚注)。
> 注意:范例的具体内容(工具名/命令/数字)是该仓库专属,**只学风格、不照抄内容**;你的文章须基于目标仓库的真实官方资料重写。

本文件只讲**怎么做**与**验收闸门**。

维护脚本(批量修正常见模板污染):[fix-beginner-sections.py](fix-beginner-sections.py)

## 工作流(五个阶段)

复制此清单并逐项推进:

```
- [ ] 1. 调研仓库:抓 README/docs/releases,搞清类型、安装、核心能力、工具清单、同类对照、风险
- [ ] 2. 定主场景:选一个贯穿全文的真实案例(如「接手多语言 monorepo」)
- [ ] 3. 套模板:按 TEMPLATE.md 逐节填充
- [ ] 4. 中文化+新手化:行话注释、名词扫盲、示例输出标注「示意」
- [ ] 5. 过闸门:逐条核对本文末「验收闸门」,不合格不交付
```

### 阶段 1 · 调研仓库(必须基于真实信息,禁止臆造)

优先用浏览器/抓取工具读官方一手资料,**不要凭记忆编造**:

- GitHub 仓库主页 + `README.md`(安装方式、核心命令、工具/API 清单)。
- 文档站、`releases`(最新版本号、安装脚本名)、必要时论文/Benchmark。
- 找出:**仓库类型**(MCP Server / CLI / 库 / App)、**安装路径**(脚本/包管理器/手动)、**核心能力 3–6 项**、**同类竞品**(用于选型对照)、**主要风险/坑**。

> 拿不准的数字(性能倍数、内存占用等)→ 注明来源或标「约/据官方」;示例输出一律标注「**示例 / 示意**」,不得伪装成真实运行结果。

### 阶段 2 · 定主场景(全文的灵魂)

选**一个**具体、可代入的真实场景贯穿始终,让每个案例都服务于它。例:
「在 Cursor 里接手 1200 文件的多语言 monorepo:先看架构 → 查调用链 → 清死代码 → 合入前看影响面」。
主场景决定案例 A–G 的取材,避免功能罗列、各说各话。

### 阶段 3–4 · 套模板 + 中文化

按 [TEMPLATE.md](TEMPLATE.md) 的节序填充。**每个案例必须凑齐「五件套」**(见下)。中文化规则见末尾闸门。

## 案例「五件套」(最关键的硬性结构)

每个 `## 案例 X` 都必须包含且**按此顺序**:

1. **`> 场景`**(blockquote):用户此刻的真实处境与痛点,一段话代入。
2. **角色**:一句话交代读者身份/前置状态。
3. **怎么用**:可直接复制粘贴给 Agent 的**那句中文话**(或命令)。
4. **你会看到什么(示例输出)**:一段 `text` 代码块,标注「示意」,让读者预期结果长什么样。
5. **效果**:`用了 vs 没用`对比(表格或要点),讲清省了什么、强在哪、怎么自检。
6. **实操卡**:固定字段的 `text` 块——
   ```text
   前置条件: ...
   可执行步骤: 1) ... 2) ... 3) ...
   验收方式: ...(怎么确认这步成了)
   失败处理: 现象 → 解决
   风险边界: ...(可逆性/注意事项)
   ```

> 「你会看到什么」和「效果」是本风格的**灵魂**,缺了就退回重写——读者要的不是步骤,是「我照做会得到什么、值不值」。

### 新手专区的硬性要素

- **名词扫盲表**:`术语 | 一句话大白话 | 生活类比`——**必须按目标仓库类型定制**,禁止从其它文章复制 MCP/索引/Token 叙事。
- **为什么要用它**:讲清**该工具**的痛点(没它时多麻烦)。
- **前置条件清单** `- [ ]`、**复制粘贴即可的安装命令**、**单独强调易漏步骤**(如"彻底重启")。
- **第一次使用的示例对话** + **成功标志** + **新手卡点表**(现象\|原因\|解决)。

### 新手专区 · 按仓库类型分叉(防模板污染)

| 类型 | 名词表必含 | 第一次使用 | 成功标志 | 禁止出现 |
|---|---|---|---|---|
| **MCP Server** | MCP、索引、tools 数量 | 「请索引这个项目」 | Settings → MCP 可见 N tools | — |
| **CLI / App** | CLI、子命令 | 案例 A 终端命令 | 命令无报错、产出符合示意 | MCP 索引叙事 |
| **Skills / 插件** | Skill、触发方式 | 案例 A「怎么用」示例话 | `/` 或 `@` 能补全 Skill | 「请索引这个项目」 |
| **Rules 文件** | Rules、加载路径 | 复制到 `.cursor/rules/` | 新对话行为符合规则 | MCP tools 数量 |
| **流程 CLI (如 Spec Kit)** | CLI、constitution、spec/plan | 按流程 Skills 顺序 | `.specify/`、`specs/` 存在 | MCP 索引叙事 |

> **反污染自检**:成稿后全文搜索 `请索引这个项目`、`Settings → MCP`、`接上 MCP 后先建地图`——若目标仓库**不是 MCP Server**,出现即退回重写新手专区。

## Obsidian 适配

- 文件名:`<工具名> 使用示例 · <一句话价值主张>.md`,放入用户指定目录(默认与同类文章同目录)。
- 用 Obsidian callout:`> [!tip]` `> [!note]` `> [!question]` `> [!warning]`。
- 站内链接用 `[[双链]]`(指向索引文章与同类对照文章)。
- 结尾加校验脚注表:`| **本文校验** | <日期>,对照 <README 链接> v<版本>(核对要点) |`。
- **版本号**:安装命令正文只写一处(新手专区 §2);案例/检查清单引用「见新手专区」或文首校验版本,避免 4 处硬编码同一 tag。
- **章节编号**:中文数字连续(一…十二);`0.x` 概览;禁止跳号(如六后直接十)。内链引用前先确认锚点存在。

## 验收闸门(逐条核对,不过不交付)

**A. 结构完整**
- [ ] 骨架表中所有 ✅ 区块齐全,章节中文数字编号连续。
- [ ] 每个案例都凑齐「五件套」(场景/角色/怎么用/示例输出/效果/实操卡)。
- [ ] 实操卡五字段齐全(前置/步骤/验收/失败处理/风险边界)。

**B. 新手可达**
- [ ] 新手专区出现:名词扫盲表(含生活类比)、复制粘贴命令、成功标志、卡点表。
- [ ] 全文行话首次出现即有中文注释(MCP、索引、调用链、Token、Cypher 等)。

**C. 真实可信**
- [ ] 安装命令、版本号、工具/命令名来自官方资料,非臆造。
- [ ] 所有示例输出标注「示例/示意」;不确定的量化指标标注来源或「约」。
- [ ] mermaid 能渲染(特殊字符已加引号),架构图标清 LLM/本地边界。
- [ ] **模板污染检查**:非 MCP 仓库不含「请索引这个项目」「Settings → MCP」「接上 MCP 后先建地图」等 MCP 专属表述。

**D. 可选型可避坑**
- [ ] 有「常见坑」表与「底层逻辑」三条。
- [ ] 有同类工具对照(若存在竞品)。
- [ ] 结尾参考链接 + 校验脚注完整。

## 反模式(避免)

- ❌ 在电梯简介后加 `> [!tip] 你是第一次接触?` 引流 callout → 新手专区本身已承担扫盲,文首保持简洁。
- ❌ 功能罗列、不绑定主场景 → 读者无法代入。
- ❌ 只有步骤、没有「你会看到什么/效果」→ 退回重写。
- ❌ 编造性能数字、安装命令、工具名 → 必须基于官方资料。
- ❌ 行话堆砌、不给大白话 → 小白劝退。
- ❌ 示例输出伪装成真实结果 → 必须标「示意」。
- ❌ 从 Codebase Memory MCP 等文章复制新手专区到非 MCP 仓库 → 按「仓库类型分叉」重写。
- ❌ 案例 C 正文漂移到 §0 之前、或章节编号跳号 → 按 TEMPLATE 节序归位。
- ❌ 同一版本号在 4+ 处硬编码 → 文首校验脚注 + 新手专区一处即可。
