# GitHub 仓库使用示例文章(房屋风格)

把一个 GitHub 开源仓库写成一篇**可直接照做**的中文使用示例长文。读者从「完全的小白(大一新生)」到「想快速选型的工程师」都能各取所需。

> 本 Skill **自包含、可移植**:分享给其他同学或其他 Agent 时,无需访问任何外部文件即可使用。
> - 骨架模板:[TEMPLATE.md](TEMPLATE.md)(逐节占位符,复制即用)
> - **黄金范例(整篇成稿)**:[EXAMPLE.md](EXAMPLE.md)(《Codebase Memory MCP 使用示例》全文,内嵌于本目录)
>
> **首次写作前,务必先读一遍 [EXAMPLE.md](EXAMPLE.md) 体会房屋风格**(节序、案例五件套、callout、语气、中文化、校验脚注)。
> 注意:范例的具体内容(工具名/命令/数字)是该仓库专属,**只学风格、不照抄内容**;你的文章须基于目标仓库的真实官方资料重写。

本文件只讲**怎么做**与**验收闸门**。

> **基线原则 · 每篇文章默认互相独立**:每篇使用示例文都是**自包含、可独立阅读**的成品,默认**不依赖也不引用**其它文章。提及索引文 / 姊妹文 / 对照文时**优先用纯文本「文章名」**,能不点名就不点名。**绝不用 `[[双链]]`**;若确需可点击的跨文引用,**用标准 Markdown 链接** `[文本](<文件名.md>)`(尖括号包裹含空格/特殊字符的路径,兼容 Obsidian / GitHub / 通用渲染器)——但能用纯文本就别加链接。锚点只在**文内**使用。

维护脚本(批量修稿):
- [fix-beginner-sections.py](fix-beginner-sections.py) — 新手专区 MCP 模板污染
- [fix-wikilink-anchors.py](fix-wikilink-anchors.py) — 把 `[[#标题]]` 批量改为 `[标题](#slug)`

## 工作流(五个阶段)

复制此清单并逐项推进:

```
- [ ] 1. 调研仓库:抓 README/docs/releases,搞清类型、安装、核心能力、工具清单、同类对照、风险
- [ ] 2. 定主场景:选一个贯穿全文的真实案例(如「接手多语言 monorepo」)
- [ ] 3. 套模板:按 TEMPLATE.md 逐节填充
- [ ] 4. 中文化+新手化:行话注释、名词扫盲、示例输出标注「示意」
- [ ] 5. 过闸门:逐条核对本文末「验收闸门」,不合格不交付
- [ ] 6. 锚点自检:全文搜索 `\[\[#` 应为 0;文内链均为 `[文字](#slug)`(见「Obsidian 适配」)
```

### 阶段 1 · 调研仓库(必须基于真实信息,禁止臆造)

优先用浏览器/抓取工具读官方一手资料,**不要凭记忆编造**:

- GitHub 仓库主页 + `README.md`(安装方式、核心命令、工具/API 清单)。
- 文档站、`releases`(最新版本号、安装脚本名)、必要时论文/Benchmark。
- 找出:**仓库类型**(MCP Server / CLI / 库 / App)、**安装路径**(脚本/包管理器/手动)、**核心能力 3–6 项**、**同类竞品**(用于选型对照)、**主要风险/坑**。

> 拿不准的数字(性能倍数、内存占用等)→ 注明来源或标「约/据官方」;示例输出一律标注「**示例 / 示意**」,不得伪装成真实运行结果。

### 阶段 2 · 定主场景(全文的灵魂)

选**一个**具体、可代入的真实场景贯穿始终,让每个案例都服务于它。例:
「在 Cursor 里接手 1200 文件的多语言 monorepo:先看架构 → 查调用链 → 清死代码 → 合入前看影响面」。
主场景决定案例 A–G 的取材,避免功能罗列、各说各话。

### 阶段 3–4 · 套模板 + 中文化

按 [TEMPLATE.md](TEMPLATE.md) 的节序填充。**每个案例必须凑齐「五件套」**(见下)。中文化规则见末尾闸门。

## 案例「五件套」(最关键的硬性结构)

每个 `## 案例 X` 都必须包含且**按此顺序**:

1. **`> 场景`**(blockquote):用户此刻的真实处境与痛点,一段话代入。
2. **角色**:一句话交代读者身份/前置状态。
3. **怎么用**:可直接复制粘贴给 Agent 的**那句中文话**(或命令)。
4. **你会看到什么(示例输出)**:一段 `text` 代码块,标注「示意」,让读者预期结果长什么样。
5. **效果**:`用了 vs 没用`对比(表格或要点),讲清省了什么、强在哪、怎么自检。
6. **实操卡**:固定字段的 `text` 块——
   ```text
   前置条件: ...
   可执行步骤: 1) ... 2) ... 3) ...
   验收方式: ...(怎么确认这步成了)
   失败处理: 现象 → 解决
   风险边界: ...(可逆性/注意事项)
   ```

> 「你会看到什么」和「效果」是本风格的**灵魂**,缺了就退回重写——读者要的不是步骤,是「我照做会得到什么、值不值」。

> **效果对比表去重(控制重叠度)**:全文设**一张「关键收益总结」master 表**(共性「用了 vs 没用」收益集中放这里)。各案例的「效果」表**只保留该案例特有的行**,并在表题注明「共性收益见 §关键收益总结」(文内锚点 `[文字](#slug)`)。避免同样的「需求阶段 / 可追溯 / 任务拆解」在每个案例与总结里重复 N 遍——既累赘又难维护。

## 文章骨架(节序)

> 章节用中文数字编号(一、二…),`0.x` 用于概览。完整带注释骨架见 [TEMPLATE.md](TEMPLATE.md)。

| 区块 | 作用 | 必须? |
|---|---|:---:|
| 标题 `<工具名> 使用示例:<一句话价值>` | 点题 | ✅ |
| 开篇 `> 电梯简介` | 一句话讲清机制→价值(给老手) | ✅ |
| `## 新手专区:从 0 到 1 手把手` | 大白话扫盲+复制粘贴上手(电梯简介后直接进入,**不加** tip 引流 callout) | ✅ |
| `## 仓库能力入口一览` | 入口关系/工具清单/CLI/宿主 | ✅ |
| `## 0. 案例背景` | 设定贯穿全文的主场景 | ✅ |
| `## 0.1 这个仓库到底是什么?` | 分层表 + **架构原理图(mermaid)** + 同类边界对照 | ✅ |
| `## 0.2 要不要一上来全量使用?` | 适合/不适合 | ✅ |
| `## 0.3 功能覆盖索引` | 功能\|位置\|第一天需要吗\|风险 | ✅ |
| `## 一~七、案例 A–G` | 核心:每个走「五件套」 | ✅(数量按仓库定) |
| `## …、如何看效果` | A/B 对比实验(编号接上一节;案例排到 §七时才用「七·五」) | 建议 |
| `## …、与同类怎么选/怎么叠` | 选型 | 建议 |
| `## …、常见坑` | 坑\|原因\|规避 表 | ✅ |
| `## …、实操检查清单` | 交付前自检 `- [ ]` | ✅ |
| `## …、底层逻辑` | 用前必记的心智模型(标题数字=实际条数) | ✅ |
| `## …、参考链接` | 官方/GitHub 外链 + 校验脚注(姊妹文/索引文用纯文本或标准 Markdown 链接,不用双链) | ✅ |

### 新手专区的硬性要素

- **名词扫盲表**:`术语 | 一句话大白话 | 生活类比`——**必须按目标仓库类型定制**,禁止从其它文章复制 MCP/索引/Token 叙事。
- **为什么要用它**:讲清**该工具**的痛点(没它时多麻烦)。
- **前置条件清单** `- [ ]`、**复制粘贴即可的安装命令**、**单独强调易漏步骤**(如"彻底重启")。
- **第一次使用的示例对话** + **成功标志** + **新手卡点表**(现象\|原因\|解决)。

### 新手专区 · 按仓库类型分叉(防模板污染)

| 类型 | 名词表必含 | 第一次使用 | 成功标志 | 禁止出现 |
|---|---|---|---|---|
| **MCP Server** | MCP、索引、tools 数量 | 「请索引这个项目」 | Settings → MCP 可见 N tools | — |
| **CLI / App** | CLI、子命令 | 案例 A 终端命令 | 命令无报错、产出符合示意 | MCP 索引叙事 |
| **Skills / 插件** | Skill、触发方式 | 案例 A「怎么用」示例话 | `/` 或 `@` 能补全 Skill | 「请索引这个项目」 |
| **Rules 文件** | Rules、加载路径 | 复制到 `.cursor/rules/` | 新对话行为符合规则 | MCP tools 数量 |
| **流程 CLI (如 Spec Kit)** | CLI、constitution、spec/plan | 按流程 Skills 顺序 | `.specify/`、`specs/` 存在 | MCP 索引叙事 |

> **反污染自检**:成稿后全文搜索 `请索引这个项目`、`Settings → MCP`、`接上 MCP 后先建地图`——若目标仓库**不是 MCP Server**,出现即退回重写新手专区。

> **CLI / 流程 CLI 额外要素(别只给一条示例命令)**:
> - **教「怎么查」而非只给一条命令**:列出官方的自检三件套(如 `--help` / `xxx list` / `xxx check`),让读者查到**自己本机版本**真实支持的参数与选项,避免照搬文档里不存在的名字。
> - **关键子命令给完整参数速查表**(参数 | 作用),覆盖常用 flag(如初始化目标、强制覆盖、脚本类型、跳过检查、预设等),而不是只演示 happy path。
> - **多代理 / 多形态对照**:若工具支持多个 AI 代理或多种安装形态(如「斜杠命令型 vs skills 模式 vs generic」),用一张表给出**每种形态的代表代理 + 命令示例**,并提示「以 `xxx list` 实测为准」。
> - **跨平台命令**:Windows 用户为主时,代码块用 PowerShell 并显式给出对应的脚本类型参数(如 `--script ps`)。

## Obsidian 适配

- 文件名:`<工具名> 使用示例 · <一句话价值主张>.md`,放入用户指定目录(默认与同类文章同目录)。
- 用 Obsidian callout:`> [!tip]` `> [!note]` `> [!question]` `> [!warning]`。
- **文章独立(默认自包含)**:每篇默认可独立阅读。提及索引文/同类对照文时**优先用纯文本「文章名」**,能不点名则不点名;**绝不用 `[[双链]]`**。若确需可点击跨文引用,用标准 Markdown 链接 `[文本](<文件名.md>)`(尖括号包裹含空格/特殊字符的路径),不要用跨文档 `.md#锚点` 形式的脆弱锚点。锚点只在**文内**使用。
- **文内锚点统一用 Markdown 原生格式** `[显示文字](#slug)`——Obsidian 阅读视图、Cursor/VSCode 预览、md2html 导出均可点击。**禁止**同文档用 `[[#标题]]` / `[[#标题\|别名]]`(Obsidian 块引用,在 Cursor 预览里点不动;成稿后全文 `\[\[#` 必须为 0)。
- **slug 推算规则**(与本系列长文文首「怎么读」表格一致;批量修复见 [fix-wikilink-anchors.py](fix-wikilink-anchors.py)):
  1. ASCII 字母转小写
  2. 去掉 `（）()·:：,，` 等标点(括号**内文字保留**)
  3. ` · ` / ` + ` / `&` → `--`
  4. 空格 → `-`
  | 标题 | slug |
  |---|---|
  | `### 选型铁律（避免重叠与冲突）` | `#选型铁律避免重叠与冲突` |
  | `### 扩展工具（正文常用 · 非九仓表内）` | `#扩展工具正文常用--非九仓表内` |
  | `## L2: 意图与对齐层 (Intent & Alignment)` | `#l2-意图与对齐层-intent--alignment` |
  | `### Brownfield vs Greenfield（术语）` | `#brownfield-vs-greenfield术语` |
  | `#### 附录 A · 一键开场白（Step 0 完成后，从新需求开始）` | `#附录-a--一键开场白step-0-完成后从新需求开始` |
  - **slug 对不上时**:在 Obsidian 对目标标题右键「复制指向标题的链接」,取 `#` 后片段;或改用下方 `<a id>` 写法。
  - **要 100% 稳/不想推算 slug** → 在目标标题上方加 `<a id="英文id"></a>`,再用 `[文字](#英文id)`。
- **锚点只能指向真实存在的标题**:每个文内引用都要对应一个真实小节标题;**不要引用「下段某某」这类没有标题的段落**(点不动、读者也对不上)。需要被引用的段落,先给它一个标题再链。
- 结尾加校验脚注表:`| **本文校验** | <日期>,对照 <README 链接> v<版本>(核对要点) |`。
- **版本号(单一来源)**:真实版本号只在三处出现——文首校验基准、新手专区 §2 安装命令、文末校验脚注;并在文首注明「全文唯一版本基准」。案例/实操卡/检查清单一律引用「见新手专区」或「文首校验版本」,禁止重复硬编码同一 tag。
- **章节编号**:中文数字连续(一…十二);`0.x` 概览;禁止跳号(如六后直接十)。`七·五` 这类「点五」编号**仅当存在对应整数节(如 §七)时**才用,否则改用连续整数。**标题里的数字必须与实际条数一致**(如「三条底层逻辑」下就必须正好三条)。内链引用前先确认锚点存在。

## 验收闸门(逐条核对,不过不交付)

**A. 结构完整**
- [ ] 骨架表中所有 ✅ 区块齐全,章节中文数字编号连续。
- [ ] 每个案例都凑齐「五件套」(场景/角色/怎么用/示例输出/效果/实操卡)。
- [ ] 实操卡五字段齐全(前置/步骤/验收/失败处理/风险边界)。
- [ ] 文内锚点能真跳:统一 `[文字](#slug)`(或 `<a id>`),全文无 `\[\[#`;每个锚点指向真实标题、无「下段某某」式悬空引用。

**B. 新手可达**
- [ ] 新手专区出现:名词扫盲表(含生活类比)、复制粘贴命令、成功标志、卡点表。
- [ ] 全文行话首次出现即有中文注释(MCP、索引、调用链、Token、Cypher 等)。

**C. 真实可信**
- [ ] 安装命令、版本号、工具/命令名来自官方资料,非臆造。
- [ ] **计数已核对**:skills / tools / 子命令 / 集成的**数量**与官方 README 一致;不确定时标「约」或「以 `xxx list` 实测为准」,**不硬写一个会随版本漂移的精确数**(常见错误:把 N 写成 N+1)。
- [ ] 所有示例输出标注「示例/示意」;不确定的量化指标标注来源或「约」。
- [ ] mermaid 能渲染(特殊字符已加引号),架构图标清 LLM/本地边界。
- [ ] **模板污染检查**:非 MCP 仓库不含「请索引这个项目」「Settings → MCP」「接上 MCP 后先建地图」等 MCP 专属表述。

**D. 可选型可避坑**
- [ ] 有「常见坑」表与「底层逻辑」(标题数字 = 实际条数)。
- [ ] 有同类工具对照(若存在竞品)。
- [ ] 结尾参考链接 + 校验脚注完整。
- [ ] **文章默认自包含**:无 `[[双链]]`、无跨文档 `.md#锚点`;姊妹文/索引文引用优先纯文本,需链接时为标准 Markdown 链接(路径含特殊字符加尖括号 `<...>`)且指向真实存在的文件(无断链)。
- [ ] **版本号 / 计数等易变值收敛到单一来源**,未在 4+ 处硬编码。

## 反模式(避免)

- ❌ 在电梯简介后加 `> [!tip] 你是第一次接触?` 引流 callout → 新手专区本身已承担扫盲,文首保持简洁。
- ❌ 功能罗列、不绑定主场景 → 读者无法代入。
- ❌ 只有步骤、没有「你会看到什么/效果」→ 退回重写。
- ❌ 编造性能数字、安装命令、工具名 → 必须基于官方资料。
- ❌ 行话堆砌、不给大白话 → 小白劝退。
- ❌ 示例输出伪装成真实结果 → 必须标「示意」。
- ❌ 从 Codebase Memory MCP 等文章复制新手专区到非 MCP 仓库 → 按「仓库类型分叉」重写。
- ❌ 案例 C 正文漂移到 §0 之前、或章节编号跳号 → 按 TEMPLATE 节序归位。
- ❌ 同一版本号在 4+ 处硬编码 → 文首校验脚注 + 新手专区一处即可。
- ❌ 用 `[[双链]]` 互链索引文或姊妹文 → 优先纯文本;需链接时改标准 Markdown 链接 `[文本](<文件名.md>)`,保持每篇默认自包含。
- ❌ 同文档用 `[[#标题]]` 做内链 → 改 `[标题](#slug)`;遗留 `\[\[#` 用 [fix-wikilink-anchors.py](fix-wikilink-anchors.py) 批量修。
- ❌ slug 手算错、或锚点指向不存在的小节/「下段某某」→ 点不动 → 对照标题重算 slug,或先给被引段落加标题 / 用 `<a id>`.
- ❌ 把 skills/工具/集成**数量**凭印象写死(如「11 个」实际 10 个) → 对照官方 README 核对,或标「约 / 以 `xxx list` 实测为准」。
- ❌ 同一组「用了 vs 没用」对比行在每个案例与总结里重复 → 设 master 表,案例只留特有行 + 锚点指回。
