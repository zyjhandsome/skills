# GitHub 仓库使用示例文章(房屋风格)

把一个 GitHub 开源仓库写成一篇**可直接照做**的中文使用示例长文。读者从「想快速上手的工程师」到「想快速选型的技术负责人」都能各取所需。

> 本 Skill **自包含、可移植**:分享给其他同学或其他 Agent 时,无需访问任何外部文件即可使用。
> - 骨架模板:[TEMPLATE.md](TEMPLATE.md)(逐节占位符,复制即用)
> - **黄金范例(整篇成稿)**:[EXAMPLE.md](EXAMPLE.md)(《Codebase Memory MCP 使用示例》全文,内嵌于本目录)
>
> **首次写作前,务必先读一遍 [EXAMPLE.md](EXAMPLE.md) 体会房屋风格**(节序、案例五件套、callout、语气、中文化、校验脚注)。
> 注意:范例的具体内容(工具名/命令/数字)是该仓库专属,**只学风格、不照抄内容**;你的文章须基于目标仓库的真实官方资料重写。
> **一切规则以本 SKILL 为准**。现有 EXAMPLE.md 是早期成稿,可能仍含**历史「新手专区」、带中文序数的小节标题、硬编码的 Stars/数量**——这些都是**已废弃写法**,只借鉴它的语气/五件套/callout/校验脚注,**不要模仿上述过时结构**;新文章按本 SKILL 的「去序数 + 自动同步徽章 + 五件套 + master 表」来写。

本文件只讲**怎么做**与**验收闸门**。

> **基线原则 · 每篇文章默认互相独立**:每篇使用示例文都是**自包含、可独立阅读**的成品,默认**不依赖也不引用**其它文章。提及索引文 / 姊妹文 / 对照文时**优先用纯文本「文章名」**,能不点名就不点名。**绝不用 `[[双链]]`**;若确需可点击的跨文引用,**用标准 Markdown 链接** `[文本](<文件名.md>)`(尖括号包裹含空格/特殊字符的路径,兼容 Obsidian / GitHub / 通用渲染器)——但能用纯文本就别加链接。锚点只在**文内**使用。

维护脚本(批量修稿):
- [fix-wikilink-anchors.py](fix-wikilink-anchors.py) — 把 `[[#标题]]` 批量改为 `[标题](#slug)`
- [fix-beginner-sections.py](fix-beginner-sections.py) — **历史脚本**:清理已废弃「新手专区」里的 MCP 模板污染(存量修稿用)

## 工作流(五个阶段)

复制此清单并逐项推进:

```
- [ ] 1. 调研仓库:抓 README/docs/releases,搞清类型、安装、核心能力、工具清单、同类对照、风险
- [ ] 2. 定主场景:选一个贯穿全文的真实案例(如「接手多语言 monorepo」)
- [ ] 3. 套模板:按 TEMPLATE.md 逐节填充
- [ ] 4. 中文化:行话首次注释、示例输出标注「示意」、安装/验收写进案例 A
- [ ] 5. 过闸门:逐条核对本文末「验收闸门」,不合格不交付
- [ ] 6. 锚点自检:全文搜索 `\[\[#` 应为 0;文内链均为 `[文字](#slug)`(见「Obsidian 适配」)
```

### 阶段 1 · 调研仓库(必须基于真实信息,禁止臆造)

优先用浏览器/抓取工具读官方一手资料,**不要凭记忆编造**:

- GitHub 仓库主页 + `README.md`(安装方式、核心命令、工具/API 清单)。
- 文档站、`releases`(最新版本号、安装脚本名)、必要时论文/Benchmark。
- 找出:**仓库类型**(MCP Server / CLI / 库 / App / Skills)、**安装路径**(脚本/包管理器/手动)、**核心能力 3–6 项**、**同类竞品**(用于选型对照)、**主要风险/坑**。

#### 使用样例从哪来(样例可溯源 · 按优先级)

案例不是编的,要能追到真实出处。按此顺序取材:

1. **README 的 Quickstart / Usage / Reference** — 第一手,最权威。
2. **`examples/` `demo/` 目录** — 官方认可的可运行样例。
3. **`tests/` 测试代码** — 每个功能的最小可运行调用,且一定跑得通(最易被忽略)。
4. **`docs/` 文档站 / 教程**。
5. **`CHANGELOG` / Releases** — 新特性常带用法片段。
6. **Issues / Discussions** — 真实场景与坑,搜 `example` / `how to` / `usage`。
7. **GitHub 代码搜索、包管理页(npm/PyPI/…)、社区博客** — 找生产级真实用法。

> **无 `examples/`、`tests/` 的仓库**(如纯 SKILL.md / Rules 仓):以 README + 各条目源文件 + 官方分发页(如 skills.sh)+ 社区为准,并在文中注明。

#### 联网核对当前真实情况(交付前必做)

- 抓一次**当前 README/Reference + 分发页**,核对:命令名/别名、条目清单、分类口径、是否有重命名/新增/移除。
- 别只信记忆或旧稿——工具名、安装命令、条目清单是高频变动项。

> 拿不准的数字(性能倍数、内存占用、Stars、安装量等)→ 用**自动同步徽章**(见「活跃度 / 指标呈现」)或注明「约/据官方/以来源页面为准」;示例输出一律标注「**示例 / 示意**」,不得伪装成真实运行结果。

### 阶段 2 · 定主场景(全文的灵魂)

选**一个**具体、可代入的真实场景贯穿始终,让每个案例都服务于它。例:
「在 Cursor 里接手 1200 文件的多语言 monorepo:先看架构 → 查调用链 → 清死代码 → 合入前看影响面」。
主场景决定案例 A–G 的取材,避免功能罗列、各说各话。

### 阶段 2.5 · 样例覆盖度(能力全集 → 覆盖矩阵)

主场景保证「连贯」,覆盖矩阵保证「不漏」。二者缺一不可。

1. **建能力全集表**:把仓库**全部**功能/命令/子模块/技能列成一张表(来自阶段 1 的目录与 Reference)。这是覆盖的**兜底基准**。
2. **建功能 × 案例覆盖矩阵**:每个案例标注它覆盖了哪些能力;做成「功能 → 案例位置 / 第一天需要吗 / 风险」表(即 §0.3 功能覆盖索引)。
3. **反查缺口**:全集表里**每一行都要能在覆盖索引里找到至少一个案例**;找不到的行 = 覆盖缺口,补案例或并入「补齐案例」。

**判「典型」的三重标准(缺一就不算典型):**

| 标准 | 含义 | 怎么满足 |
|---|---|---|
| **功能覆盖典型** | 每个能力各有代表性用法 | 覆盖矩阵无缺口;**高频/官方头号推荐的能力必须有案例**,别被主线故事挤掉 |
| **真实工作流典型** | 案例串成真实端到端流程 | 用一条主场景按「安装→…→收尾」顺序贯穿 |
| **可溯源典型** | 能对到 README/examples/tests/社区 | 每个案例的用法都追得到出处(见阶段 1 来源) |

> **高频能力不能漏**:用分发页/社区热度判断哪些是最常用能力(如安装量、README 头号推荐)。**最常用的能力若没有案例,就是最大的覆盖缺口**——即使它不完美契合主线故事,也要给它一个 mini 案例或明确定位。

### 阶段 3–4 · 套模板 + 中文化

按 [TEMPLATE.md](TEMPLATE.md) 的节序填充。**每个案例必须凑齐「五件套」**(见下)。中文化规则见末尾闸门。

## 案例「五件套」(最关键的硬性结构)

每个 `## 案例 X` 都必须包含且**按此顺序**:

1. **`> 场景`**(blockquote):用户此刻的真实处境与痛点,一段话代入。
2. **角色**:一句话交代读者身份/前置状态。**必须区别于「场景」**——不得逐字复制上一行 `> 场景` blockquote(常见翻车:两行内容一模一样)。
3. **怎么用**:可直接复制粘贴给 Agent 的**那句中文话**(或命令)。
4. **你会看到什么(示例输出)**:一段 `text` 代码块,标注「示意」,让读者预期结果长什么样。
5. **效果**:`用了 vs 没用`对比(表格或要点),讲清省了什么、强在哪、怎么自检。
6. **实操卡**:固定字段的 `text` 块——
   ```text
   前置条件: ...
   可执行步骤: 1) ... 2) ... 3) ...
   验收方式: ...(怎么确认这步成了)
   失败处理: 现象 → 解决
   风险边界: ...(可逆性/注意事项)
   ```

> 「你会看到什么」和「效果」是本风格的**灵魂**,缺了就退回重写——读者要的不是步骤,是「我照做会得到什么、值不值」。

> **效果对比表去重(控制重叠度)**:全文设**一张「关键收益总结」master 表**(共性「用了 vs 没用」收益集中放这里)。各案例的「效果」表**只保留该案例特有的行**,并在表题注明「共性收益见 §关键收益总结」(文内锚点 `[文字](#slug)`)。避免同样的「需求阶段 / 可追溯 / 任务拆解」在每个案例与总结里重复 N 遍——既累赘又难维护。

## 文章骨架(节序)

> **正文小节标题默认不带中文序数**(不写「一、二、七、」)。顺序靠**文档先后 + 案例字母(案例 A–G)**表达;只有 `0 / 0.1 / 0.2 / 0.3` 概览簇保留数字(稳定、极少重排)。
> **为什么去序数**:插入/重排小节时不必全篇重编号,不会跳号,交叉引用也不会失效;中文带序数的标题生成的锚点 slug 还不可靠。
> **交叉引用**:一律用**名称或案例字母**(如「见『常见坑』」「见案例 A」);能纯文本就别加锚点链接,必要时才用 `[常见坑](#常见坑)`(命名标题 slug 更稳)。
> 完整带注释骨架见 [TEMPLATE.md](TEMPLATE.md)。

| 区块 | 作用 | 必须? |
|---|---|:---:|
| 标题 `<工具名> 使用示例:<一句话价值>` | 点题 | ✅ |
| 开篇 `> 电梯简介` | 一句话讲清机制→价值 | ✅ |
| 自动同步徽章一行(Stars/release/安装量) | 见「活跃度 / 指标呈现」 | 建议 |
| `## 仓库能力入口一览`(或 `仓库 Skills 一览`) | **整仓作用与原理** + 入口/工具清单/CLI/宿主 | ✅ |
| `## 0. 案例背景` | 设定贯穿全文的主场景 | ✅ |
| `## 0.1 这个仓库到底是什么?` | 可瘦身为回链;详述已在「整仓作用与原理」 | 建议 |
| `## 0.2 要不要一上来全量使用?` | 适合/不适合 | ✅ |
| `## 0.3 功能覆盖索引` | 功能\|位置\|第一天需要吗\|风险(= 覆盖矩阵,见阶段 2.5) | ✅ |
| `## 案例 A–G · <标题>` | 核心:每个走「五件套」;**案例 A 承担安装+首次验收** | ✅(数量按仓库定) |
| `## 关键收益总结(用了 vs 没用)` | 共性收益 master 表;各案例「效果」只留特有行并回指此表 | ✅ |
| `## 如何看效果` | A/B 对比实验 | 建议 |
| `## 与同类怎么选/怎么叠` | 选型 | 建议 |
| `## 常见坑` | 坑\|原因\|规避 表(含易漏步骤如重启) | ✅ |
| `## 实操检查清单` | 交付前自检 `- [ ]` | ✅ |
| `## 底层逻辑` | 用前必记的心智模型(标题不写死条数) | ✅ |
| `## 参考链接` | 官方/GitHub 外链 + 校验脚注 | ✅ |

### 「整仓作用与原理」硬性要素(放在能力入口一览**最前**)

- **一句话电梯**:机制 → 价值 → 与流程主人类工具的边界。
- **解决什么问题**:README 式问题列表或对照表(问题 → fix → 代表技能/命令)。
- **运行机制**:架构原理 **mermaid**;标清 LLM / 本地 / 双轴触发等边界。
- **典型链路 mermaid**:install → 首次配置 → 核心工作流(可与 §0.2 最小路径呼应)。
- **与邻工具边界表**(若存在竞品)。
- **类型专属配置子节**(若需要):如 Skills 的 `setup` 每仓初始化、MCP 的索引约定——**嵌在此节末尾**,不单开「新手专区」。

### 案例 A · 安装与首次使用(按仓库类型)

| 类型 | 案例 A 必含 | 验收方式示例 | 禁止出现(非 MCP 仓) |
|---|---|---|---|
| **MCP Server** | 索引命令、`tools` 数量来源 | Settings → MCP 可见 tools | — |
| **CLI / App** | 终端命令 + `--help`/`list`/`check` 自检 | 命令无报错、产出符合示意 | 「请索引这个项目」 |
| **Skills / 插件** | `npx skills add` 等 + `/setup` + 重启宿主 | `/` 或 `@` 能补全 Skill | MCP 索引叙事 |
| **Rules 文件** | 复制路径 + 加载规则 | 新对话行为符合规则 | MCP tools 数量 |
| **流程 CLI (Spec Kit)** | 流程顺序 + constitution/spec | `.specify/`、`specs/` 存在 | MCP 索引叙事 |

> **反污染自检**:成稿后全文搜索 `请索引这个项目`、`Settings → MCP`、`接上 MCP 后先建地图`——若目标仓库**不是 MCP Server**,出现即退回重写。

> **CLI / 流程 CLI 额外要素(别只给一条 happy-path)**:
> - 列出官方自检三件套(`--help` / `xxx list` / `xxx check`)。
> - 关键子命令参数速查表(参数 \| 作用)。
> - 多代理 / 多形态对照表 + 「以 `xxx list` 实测为准」。
> - Windows 为主时用 PowerShell 并给 `--script ps` 等。

### 活跃度 / 指标呈现(自动同步徽章 · 首选)

**Stars / Forks / release / 下载量 / 安装量 这类高波动数字,绝不硬编码进正文**(写死当天就过时,还会 N 处漂移)。优先用**能自动同步的徽章**呈现——源头一变,徽章自动更新,无需人工维护:

- **位置**:放在**文章标题的引言(电梯简介)之后、正文表格之前**,单独一行。
- **GitHub 仓库(Stars / release)**:

```text
[![GitHub stars](https://img.shields.io/github/stars/{owner}/{repo}?style=social)](https://github.com/{owner}/{repo}) [![GitHub release](https://img.shields.io/github/v/release/{owner}/{repo})](https://github.com/{owner}/{repo}/releases)
```

- **总安装量/下载量(若源头提供徽章)**:优先用官方徽章而非手写。例如 **skills.sh 总安装量**有官方徽章,自动同步:

```text
[![skills.sh](https://skills.sh/b/{owner}/{repo})](https://skills.sh/{owner}/{repo})
```

  其它平台同理:npm 用 `img.shields.io/npm/dm/{pkg}`、PyPI 用 `img.shields.io/pypi/dm/{pkg}` 等。

- **正文只写定性**:「活跃度信号」「校验脚注」等处**只写定性判断**(如「高活跃/官方头号推荐」),并注明「**定量见上方徽章 / 以来源页面为准**」;**不再手写** `⭐ 79k+ Stars` 这类会漂移的精确数。
- **逐条目指标(如每个技能/命令的安装量)通常没有单独徽章 → 一律不硬编码**,改用定性词(「最常用 / 头号推荐 / 常用 / 较新」)。判断热度用于**选案例**(阶段 2.5),但不落成会漂移的数字。
- **非 GitHub / 无公开指标的商业产品**:不加徽章,活跃度写「商业产品 · 以官网/后台为准」。

> 版本号仍遵守「单一来源」:release 徽章已呈现 latest,正文/校验脚注不必再硬写同一 tag(需要点明某个里程碑版本时才写一次)。

## Obsidian 适配

- 文件名:`<工具名> 使用示例 · <一句话价值主张>.md`,放入用户指定目录(默认与同类文章同目录)。
- 用 Obsidian callout:`> [!tip]` `> [!note]` `> [!question]` `> [!warning]`。
- **文章独立(默认自包含)**:每篇默认可独立阅读。**绝不用 `[[双链]]`**;跨文引用用标准 Markdown 链接 `[文本](<文件名.md>)`。锚点只在**文内**使用。
- **文内锚点**: `[显示文字](#slug)`;成稿后全文 `\[\[#` 必须为 0。批量修复见 [fix-wikilink-anchors.py](fix-wikilink-anchors.py)。
- **版本号(单一来源)**:真实版本号只在**案例 A 安装命令**与**文末校验脚注**两处硬编码;其它章节写「见案例 A」或「见校验脚注」。
- **章节编号(去序数)**:正文小节标题**不带中文序数**;仅 `0.x` 概览簇与 案例 A–G 字母保留。交叉引用按**名称/案例字母**,不按序数。这样插入/重排小节无需全篇重编号,也不会有跳号或失效引用。

## 验收闸门(逐条核对,不过不交付)

**A. 结构完整**
- [ ] 骨架表中所有 ✅ 区块齐全;正文小节标题**无中文序数**(仅 `0.x` 与 案例 A–G),交叉引用按名称/字母、无失效引用。
- [ ] 每个案例都凑齐「五件套」(场景/角色/怎么用/示例输出/效果/实操卡),且**角色不与场景重复**。
- [ ] 有一张「关键收益总结」master 表;各案例「效果」只留特有行并回指总表。
- [ ] 实操卡五字段齐全(前置/步骤/验收/失败处理/风险边界)。

## 终检(交付前最后一遍 · 建议用搜索逐条扫)

结构改动(尤其插入/重排小节、增删条目)后,过一遍这张「回归清单」:

- [ ] **序数残留**:全文搜 `^## [一二三四五六七八九十]` 应为 0(正文小节已去序数)。
- [ ] **失效引用**:搜历史序数引用(如 `五、常见坑`)与旧锚点,确认都已改为名称/字母引用。
- [ ] **易漂移数字**:搜 `Stars`、`K 安装`、`M 安装`、`下载` 等,精确数应改为徽章或定性 + 「以来源页面为准」。
- [ ] **角色 = 场景**:逐个案例比对「> 场景」与「角色」两行,不得雷同。
- [ ] **五件套完整**:每个案例都有 怎么用 / 你会看到什么(标「示意」)/ 效果。
- [ ] **覆盖矩阵无缺口**:能力全集每行都能在 §功能覆盖索引 找到案例;高频能力有案例。
- [ ] **示意标注**:所有示例输出块都标了「示例 / 示意」。
- [ ] **反污染**:非 MCP 仓无「请索引这个项目」「Settings → MCP」等 MCP 叙事。
- [ ] **错别字/夹带**:通读一遍,无跨语言乱码、无占位符残留。

### 9 维度评分鱼骨(交付前自评 / 评审打分)

逐维打分(建议 ≥8 才交付),低于 8 的维度先修再交:

| 维度 | 看什么 | 高频扣分点 |
|---|---|---|
| **文件名** | `<工具名> 使用示例 · <价值主张>.md`,与同类命名一致 | 插入冗余括号、与姊妹文风格不一 |
| **逻辑性** | 小节顺序合理、无跳号、引用不失效 | 序数断裂、交叉引用指错 |
| **清晰性** | 原理图 + 双轴/边界 + 索引清楚 | 术语无注释、堆砌 |
| **准确性** | 命令/条目/分类对齐当前官方,已联网核对 | 凭记忆、别名过时 |
| **时效性** | 校验日期 + 徽章 + 高波动字段标注 | 硬编码 Stars/安装量 |
| **实用性** | 五件套齐全、可照做、能预览结果 | 只有步骤没有「看到什么/效果」 |
| **易用性** | 锚点/索引好用、导航顺 | 引用按序数、slug 不可靠 |
| **重叠度** | master 表去重、角色≠场景 | 效果对比逐案例重复、角色复制场景 |
| **可维护性** | 数字收敛到单一来源/徽章、去序数免重排 | 多处硬编码、插节要全篇重编号 |

## 反模式(避免)

- ❌ 单独开「新手专区」重复案例 A / 整仓原理 → 安装与验收写进**案例 A**,原理写进**整仓作用与原理**。
- ❌ 在电梯简介后加冗长引流 callout → 文首保持简洁,电梯后直接进能力入口一览。
- ❌ 功能罗列、不绑定主场景 → 读者无法代入。
- ❌ 只有步骤、没有「你会看到什么/效果」→ 退回重写。
- ❌ 编造性能数字、安装命令、工具名 → 必须基于官方资料,并联网核对当前状态。
- ❌ 示例输出伪装成真实结果 → 必须标「示意」。
- ❌ 从 MCP 范例复制索引叙事到 Skills/CLI 仓库 → 按「案例 A 类型分叉」重写。
- ❌ 同一版本号在 4+ 处硬编码 → 案例 A + 校验脚注两处即可。
- ❌ 手写 `⭐ Nk+ Stars` 或**逐条目安装量/下载量** → 用自动同步徽章(含 skills.sh 徽章);逐条目改用定性词。
- ❌ 用 `[[双链]]` 互链其它文章 → 标准 Markdown 链接或纯文本。
- ❌ 同一组「用了 vs 没用」对比行在每个案例重复 → master 表 + 案例只留特有行。
- ❌ CLI 类只给一条 happy-path → 补自检三件套 + 参数速查表。
- ❌ **「角色」逐字复制「场景」** → 角色只写读者身份/前置状态,与场景不重复。
- ❌ **正文小节标题带中文序数**(一、二、七…) → 去序数;顺序靠文档先后 + 案例字母,引用按名称。
- ❌ **只覆盖主线顺手的能力、漏掉高频能力** → 用覆盖矩阵反查缺口,高频/头号推荐能力必须有案例(见阶段 2.5)。
