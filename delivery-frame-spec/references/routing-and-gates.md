# 路由与闸门参考

## 路由决策

```text
是否要求修改？
├─ 否 → 只读
└─ 是
   ├─ 失败/回归？→ 调试
   ├─ 已有批准且可执行的任务清单？→ delivery-execute-verify
   ├─ 已有批准的规格？→ delivery-plan-tasks
   ├─ 局部影响可限定且无红线信号？→ 快速
   ├─ 命中红线或多个升级信号？→ 高风险
   └─ 其他情况 → 标准
```

无法判断时选择更安全的较高车道，并说明什么证据可以支持降级。

## 快速车道负例（禁止判 Quick）

下列任一出现 → **不得**选 Quick（通常 High；仅当代码事实证明未触及该面时可 Standard，且 Risk Rating 必须写清 miss 理由）：

| 负例 | 为何不是 Quick |
|---|---|
| 修改鉴权策略、登录态、角色/权限判断 | 红线：auth/permission |
| 支付、计价、退款、额度扣减 | 红线：payment |
| 个人信息存储、导出、日志落敏、同意撤回 | 红线：privacy |
| 数据 schema / 迁移 / 回填 | 红线：migration |
| 公共 API/协议字段或兼容性破坏 | 红线：public contract |
| 删除数据、不可逆清理、破坏性运维动作 | 红线：destructive |
| 核心下单/结算/授权主路径行为变更 | 红线：core business |
| 「只改一小文件」但文件是共享鉴权中间件 | 影响面不可当局部 |

**Risk Rating 写法（Standard/High 必填）：** 命中特征 / 未命中高风险特征 / 不确定点 / 闸门建议。只写「Low/Quick」标签而无 hit/miss → 闸门失败。

## 快速车道轻量契约

写入活跃 OpenSpec change 的 `proposal.md`。含 explore handoff 时必须带下方勾选块（与 Standard 简报同标题）；无则写 `N/A — 无 explore handoff`。

```text
目标：
非目标：
影响文件/符号：
可观察行为：
最小验证：
禁止范围：
风险/未知项：
澄清完整性扫描：已检查的适用维度 / 证据已解决项 / 新增开放问题 / N/A 或不在范围项 / 是否仍阻塞
（可选）开放问题：

### Explore 交接消费
- [ ] chosen_direction → 已写入目标（或用户修订）
- [ ] non_goals → 已写入非目标 / N/A
- [ ] code_anchors → 已检查并引用路径或符号
- [ ] risk_signal → 仅线索；风险已按代码事实重算
- [ ] unknowns → 已写入开放问题或非阻塞丢弃
（无 explore handoff：N/A — 无 explore handoff）

实施批准：待批准 / 已批准（批准人 + 时间）
```

**通用实施批准**：任何车道开始实施前都必须获得用户显式放行。快速 / 调试-低风险 = 用户批准本契约；标准（中风险）/ 高风险 = `delivery-plan-tasks` 的实现闸门。契约写完 ≠ 可以开工；「实施批准」未记录为已批准时，禁止交接给 `delivery-execute-verify` 执行变更。

**Quick 收尾形状（防历史债务回潮）：** execute 阶段仍须产出 `verification.md`（命令/时间/结果）。勿把缺少 verification 的旧 polish archive 当作模板。

**对用户的闸门问法（易用性）**：规格闸门与实现闸门都是**单题放行**（见 `batch-clarification.md` 的 Gate ask shape）。High 的 design/tasks/rollback/security/validation 是 Agent **内部**就绪清单并写入状态源，禁止展开成用户五面问卷。产品澄清（Open Questions）仍可按波多题，与门闩分开。

## 标准 / 高风险：事实就绪闸门

在写增量规格或开始产品消歧前，按简报**标题名**检查三块结构：

| 检查项 | 通过条件 |
|---|---|
| 意图 | 目标、成功标准、边界/非目标已写明 |
| 代码事实（含证据表） | 有与需求相关的现状、复用/扩展/冲突、必选挂载点、波及线索；关键结论有证据 |
| 开放问题清单 | 已写出且将对用户可见（产品决策权威台账） |
| 风险定级与闸门建议 | 含命中特征 / 未命中高风险特征 / 不确定点 / 闸门建议 |
| Explore 交接消费 | 有 explore handoff：五条均勾选；无：`N/A — 无 explore handoff` |
| 状态源与工件位置 | 唯一状态源路径明确（批准记录落此处） |

- **通过**：向用户展示简报（或含完整「开放问题清单」的精简视图），**立即**开始批量澄清（第一波：一次列全当前互不依赖的阻塞题；每题必有推荐；点击为主、输入兜底；无「全部按推荐」）；**不等待**口头「确认简报」。
- **未通过或用户质疑事实/挂载/波及**：定点补查或提一个事实向问题，更新同一简报后重检。
- **「开放问题清单」中的产品选择**：本闸门不代替；必须在对齐循环中交互澄清。

## 风险定级写法（「风险定级与闸门建议」）

| 规则 | 行为 |
|---|---|
| 无法判断 | 按中等 |
| 判断不确定 | 按更高 |
| 命中任一高风险特征 | 默认高风险，除非代码事实表明未触及该面 |
| 只写标签不写理由 | 闸门失败 |

高风险特征参考：鉴权、支付、权限、隐私、数据迁移、并发、核心业务链路、公共接口/协议、破坏性操作。

## 状态与闸门记录

唯一后端是 OpenSpec：在 change 内用 `proposal.md`（含完整简报三块）+ 增量规格承载工件，闸门状态记录在 `proposal.md` 的「状态源与工件位置」并在对话交接中报告，不再写独立状态文件。

## 闸门失败处理

若事实就绪或规格闸门失败：

1. 列出失败检查项及证据。
2. 判断缺少的是事实、意图还是批准。
3. 若事实可由代码决定，执行一次定点调查；若必须由人做产品取舍，做批量澄清（见 `batch-clarification.md`：分波列出互不依赖的阻塞题；点击为主、输入兜底；依赖树根部优先；条件题按触发再开下一波；**禁止**「全部按推荐」；**每题必须给出推荐值与依据**，推荐项标「建议：」），由用户对本波逐项明确选择。
4. 更新现有工件，绝不另建竞争副本。
5. 重新运行闸门检查。
