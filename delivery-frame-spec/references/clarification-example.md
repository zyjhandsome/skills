# 澄清话术示例（批量澄清）

阻塞产品澄清走 **批量澄清（Batch clarification）**：**点击为主，输入兜底**。同一波一次列全互不依赖的阻塞题；有依赖则分波。详见 `batch-clarification.md`。

**禁止**「全部按推荐」/ bulk-accept：列出的每道阻塞题都必须由用户明确选择（或标明不适用而跳过）。

**必须推荐**：每道固定选择题都要同时有——背景、选项（含影响）、以及明确推荐值 + 一句依据；推荐项选项加「建议：」前缀。推荐≠决策。

流程（详见 `SKILL.md` 的 Batch clarification 规则）：

1. 先展示完整「开放问题清单」（本波将问哪些、略去哪些非阻塞/未触发主题）。
2. 依赖树根部优先组波：本波只放入当前互不依赖的阻塞题，一次列全（优先点击 UI；否则 Markdown，允许 `1B 2A` 一次回多题）；每题都带推荐。
3. 用户作答后记录；条件题仅在触发后再进入下一波。
4. 无硬数量预算；只保留会实质改变范围/验收/风险/不可逆行为的问题。

## 分波示例

### 第 1 波（互不依赖，一次列全）

开放问题清单摘要：本波问「跨重启保留」「导出重名」；「历史上限」依赖跨重启=是，本波不列入。

```text
请对本波逐项选择（可回复如：1B 2A）。每题已给出推荐，请仍逐项确认。

### 1. 转写历史是否跨重启保留？
背景：当前记录只存在内存中；跨重启保留会新增持久化、删除和隐私边界。
推荐：B — 「历史」若是正式产品能力，应跨重启；规格中会写保留上限与删除行为。
- A. 仅当前会话保留 — 范围最小，无迁移；重启后历史消失
- 建议：B. 跨重启保留 — 体验完整；需存储、删除、迁移与隐私验证
- 其他（我来说明）

### 2. 导出文件重名时的行为？
背景：导出目录固定时重名概率高；当前代码无覆盖确认逻辑。
推荐：A — 与现有下载类功能的既有行为一致。
- 建议：A. 自动追加序号 — 无打断，不丢数据
- B. 弹窗确认覆盖 — 用户可控，多一次交互
- 其他（我来说明）
```

### 第 2 波（仅当第 1 题选 B 时）

```text
### 1. 历史保留上限策略？
背景：无上限会让存储无界增长，影响启动扫描耗时。
推荐：A — 现有代码无定时任务基础设施，固定条数改动面最小。
- 建议：A. 固定条数（如 500） — 实现简单，可预期
- B. 固定时长（如 90 天） — 更符合隐私直觉，需定时清理
- 其他（我来说明）
```

（略去的非阻塞主题示例：SRT 时间戳精度——已按毫秒默认记录在案。）

## 门闩示例（对用户单问；非产品澄清）

规格闸门与实现闸门**不要**做成五面问卷。Agent 内部检查通过后，只问放行：

```text
可以开始按 tasks.md 实施了吗？
- 范围：音频文件转写 + 轻量翻译（方案 A）
- 已就绪：design/tasks；无阻塞项
- 代价：安装包含 ffmpeg；翻译开启后文本发往用户配置的 LLM
- 回滚：关掉翻译/不用文件入口即可；空闲听写路径不变
- 下一步：delivery-execute-verify，从任务 1.1 起
推荐：开始实施 — 就绪审查无阻塞，规格已批。
- 建议：开始实施
- 先不实施
- 有修改（说明）
```
