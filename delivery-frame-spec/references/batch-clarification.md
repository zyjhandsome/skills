# Batch clarification（delivery 家族共用）

固定选择题走 **批量澄清（Batch clarification）**：**点击为主，输入兜底**。本文件是四个 `delivery-*` skill 的唯一提问规范；各 skill 引用本路径，勿再复制一份互相漂移的规则。

权威路径（相对其他 skill）：`../delivery-frame-spec/references/batch-clarification.md`

本规范不绑定任何 IDE、编码代理或具体工具名。探测当前会话是否提供可点选的问卷/选择 UI；有则优先点击，无则 Markdown 输入兜底。不要为了获得点击 UI 而切换模式或改宿主设置。

需要把本规范映射到当前宿主的具体问题工具时，读取 `question-ui-adapters.md`；适配器只负责传输，不得改变本文件的分波、推荐、决策归属或停止规则。

## Family invariants

1. **分波批量**：同一波内，把当前所有**互不依赖**的阻塞固定选择题一次列全；用户一次作答。有依赖/条件的题留到触发条件满足后再开下一波——不要为依赖链假装「无条件一次列全」。
2. **禁止多题 bulk-accept**：一波含多道阻塞题时，不得提供「全部按推荐」「一次采纳」等跳过逐题确认的路径；每题都要有明确选项（或明确跳过「不适用」）。一波只有一题时，用户明确回复「按推荐」可视为选择该题的推荐项。
3. **必须给出推荐**：每一道固定选择题都要同时具备——题干/背景、2–4 个选项（含影响）、以及明确的**推荐值**与一句依据。推荐写在题干（如「推荐：B — …」），并在对应选项标签上加「建议：」前缀，便于点选辨认。推荐≠决策：不得因写了推荐就替用户作答或静默套用。
4. **2–4 个选项**；必要时加一个「其他（我来说明）」。无法诚实枚举时改自由文本，不硬凑选项（自由文本题可不标「建议：」选项，但仍应在题干说明你倾向什么输入形态，若有）。
5. **实质影响门槛**：只问会实质改变范围、验收、风险或不可逆行为的问题。无硬数量预算；凑数题禁止。
6. **点击为主，输入兜底**：优先用宿主提供的点选 UI 承载本波全部题目；不可用时用下方 Markdown 形态，允许一次回复多题（如 `1B 2A 3C`）。
7. **按就绪度停止**：把不确定性落成具体的缺失事实或待定决策：事实由 Agent 调查，实质产品决策才问用户；无实质阻塞项即停止。

## What to batch vs ask alone

| 类别 | 行为 |
|---|---|
| 产品澄清 / Open Questions | 按依赖分波；每波一次列全互不依赖的阻塞题；每题必有推荐（产品分叉仍可多题；与门闩不同） |
| 探索方向 | 看完地图后通常一题定方向；若仍有多个**互不依赖**的分叉，则一次列全那些分叉；选项中标注建议方向 |
| 方案选择（plan） | 通常一题；若同暂停点还有其他互不依赖的固定选择，可并入同一波；每题必有推荐 |
| 规格闸门 / 实施门闩 / 危险操作 | 与产品澄清分开问。对用户**只问一个放行决策**（须标推荐）。Agent **内部**仍跑完整检查清单并写入状态源；**禁止**把内部检查项（如 High 的 design/tasks/rollback/security/validation 五面）展开成让用户逐项点「通过」的问卷。High 在单问里用短摘要带上范围、代价/风险、回滚要点与下一步；Medium/High 用户选项形态一致：`建议：开始…` / `先不…` / `有修改（说明）` |

## Stage decision ownership（对齐责任）

| 阶段 | 只在本阶段向用户确认 | 不在本阶段决定 |
|---|---|---|
| `delivery-explore` | 候选方向及会改变方向选择的独立分叉 | 具体范围、验收、技术方案和实施授权；作为 `unknowns` 交给后续阶段 |
| `delivery-frame-spec` | 产品范围、可观察行为、验收、风险边界和不可逆产品决策 | 可由证据查明的事实；纯技术实现细节 |
| `delivery-plan-tasks` | 对用户约束有实质影响的技术取舍：成本/部署、迁移/兼容、回滚、运维、安全姿态或不可逆选择 | 改变产品行为/范围/验收的选择回 frame；规格内低风险可逆实现细节由 Agent 决定并记录 |
| `delivery-execute-verify` | 本阶段授权与例外：缺失实施 go、降级接受、用户改动冲突、危险/不可逆外部操作、确有成本或隔离影响的模式覆盖 | 产品疑惑回 frame；设计/任务疑惑回 plan；可由证据确定或规格内实现细节由 Agent 处理 |

**不重复确认：** 权威工件及适用条件未变化时，复用已记录的明确决定，不要求用户再次选择。阶段闸门是对下一阶段的新授权，不是重问既有决策。若后续证据使决定失效，先展示变化证据、受影响工件和需要重新决定的具体事项，再进入澄清。

## Delivery shape（每波）

1. 展示完整开放问题/决策清单（本波将问哪些；条件未触发或非阻塞略去哪些）。
2. 一次列出本波全部互不依赖的阻塞题（点击 UI 或 Markdown）；**每题含推荐**。
3. 记录每题答案；跳过未触发的条件题；若答案产生新的阻塞决策，开下一波。
4. 在宣布澄清完成前运行下方**澄清完整性扫描**；若发现新的阻塞决策则开下一波，否则停止并把其余项记为非阻塞默认假设。

## Clarification completeness sweep（Agent 内部）

在任何变更车道宣布“无阻塞问题”前执行一次；即使一开始不需要向用户提问也必须执行。只检查本次实际适用的维度，不把清单机械展开成用户问卷：

- 使用者、权限角色与入口前置条件；
- 正常状态变化，以及空值、边界值、重复操作和并发（适用时）；
- 失败、中断、重试、部分成功和恢复行为；
- 数据保存、删除、迁移、隐私与保留期限（适用时）；
- 既有调用方、公共契约、向后兼容性和部署影响；
- 性能、安全、可观测性及可证伪的验收/回归验证。

对每个适用维度执行以下处理：

1. 能由代码、测试、日志、配置或批准工件确定 → Agent 定点调查并记录事实，不问用户。
2. 会实质改变当前阶段所负责的范围、验收、风险、用户约束或不可逆行为，且必须由人决定 → 写入当前阶段的权威决策台账，按依赖进入下一波。
3. 不适用、明确不在范围或仅为可逆实现细节 → 记录为 `N/A`、out of scope 或非阻塞默认，不追问。

扫描结果必须落入当前阶段的权威 brief/contract/design/readiness/backflow 工件，明确已解决项、新增待决项、不适用项和剩余阻塞状态。Execute 发现产品或设计待决项时只生成结构化回流，不在执行阶段代替上游决定。

## Question fields（语义层）

| 语义 | 用途 |
|---|---|
| `prompt` | 题干：背景一句（代码事实）+ 问什么 + **推荐：&lt;选项&gt; — &lt;一句依据&gt;** |
| `options[]` | 2–4 项；每项短标签 + 一句影响；**推荐项标签加「建议：」前缀** |
| `recommendation` | 必填（固定选择题）：指向 `options[]` 之一 + 依据；与题干「推荐：」及选项前缀一致 |
| `allow_other` | 需要时加「其他（我来说明）」 |
| `applies_when` | 条件题注明「仅当…才适用」；本波未触发则不要列入 |

## Markdown fallback（点击 UI 不可用）

一次列出本波全部题目。示例：

```text
请对本波问题逐项选择（可回复如：1B 2A）。不适用的条件题请写「跳过」。
每题已给出推荐；多题时请仍逐项确认（不要回复「全部按推荐」）。

### 1. <问题标题>？
背景：<一句代码事实>
推荐：B — <一句依据>
- A. <选项> — <影响>
- 建议：B. <选项> — <影响>
- 其他（我来说明）

### 2. <问题标题>？（仅当第 1 题选 B 时适用 — 若本波未并列请忽略）
背景：...
推荐：A — <一句依据>
- 建议：A. <选项> — <影响>
- B. <选项> — <影响>
...
```

## Gate ask shape（对用户）

**规格闸门**（`delivery-frame-spec`，阻塞产品题已决且规格已写）：一题即可。

```text
可以按当前范围进入设计/任务分解吗？
- 范围：<一句>
- 已就绪：简报 + 增量规格；阻塞开放问题已决
- 下一步：delivery-plan-tasks 写 design/tasks（仍不写实现代码）
推荐：批准范围 — …
- 建议：批准范围并进入设计
- 先不进入（说明）
- 要改规格（说明）
```

**实现闸门**（`delivery-plan-tasks`，就绪审查无阻塞项）：一题即可。Medium 与 High **同一问法**；High 多 2–4 行代价/回滚摘要。Agent 须先在状态源记录内部五面（design/tasks/rollback/security/validation）均已自检通过，再发本问。

```text
可以开始按 tasks.md 实施了吗？
- 范围：<一句>
- 已就绪：design/tasks；无阻塞项
- 代价/风险：<High 必填短摘要；Medium 可极简>
- 回滚：<一句人话，避免只写「可逆」>
- 下一步：delivery-execute-verify，从任务 <id> 起
推荐：开始实施 — …
- 建议：开始实施
- 先不实施
- 有修改（说明）
```

用户答「开始实施 / 批准范围」= 放行该门闩整包（含 Agent 已自检的内部面），**不是**「全部按推荐」跳过未展示的产品澄清题。

## Anti-patterns

- 绑定或点名特定 IDE / 编码代理 / 选择工具。
- 为提问而切换模式或改宿主配置。
- 把有依赖的题与其前提题强行同一波「假装独立」。
- 多题波次提供「全部按推荐」，或静默套用推荐/默认。
- 只给选项、不写推荐值与依据（或只在题外含糊说「我倾向…」却不标到具体选项）。
- 为显得严谨而堆砌无实质影响的问题。
- 把内部闸门检查清单（尤其 High 五面）展开成用户逐项问卷。
- 把本规范抄进 OpenSpec 简报正文。
